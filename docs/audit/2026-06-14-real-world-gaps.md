# NamiDB real-world gap audit (2026-06-14)

## Summary

This document covers a verification pass over 16 limitations reported from a real bulk-load plus ABAC workload, plus 2 storage-portability items on S3 conditional puts (18 records total). Each was checked against the actual 0.17.1 source with file:line evidence, competitor handling, and a proposed fix. Verdict breakdown: 8 `confirmed-real`, 7 `partially-real` (a real gap, but narrower, wider, or mis-scoped vs the report), 2 `already-fixed` on 0.17.1 (`01-unwind-bind` and `10-double-identity`, both shipped before HEAD), and 1 `cannot-reproduce` (`s3a`, against the pinned `object_store` 0.13.2 whose S3 conditional-put default is already `ETagMatch`). So 15 of the 18 are still live work in some form. The biggest risks are the bulk-load blockers (`02-set-plus-map`, `03-bulk-ingest`) and the one silent-correctness hole (`09-uniqueness`, where the Python bulk path commits duplicate unique-property values with no error). The single highest-leverage move is `02-set-plus-map`: it is a confirmed half-wired feature where parser, AST, planner, and all executor helpers already exist, so wiring the one missing executor arm closes the standard Cypher upsert idiom (`MERGE ... ON MATCH SET n += $props`) at S effort, which in turn unblocks the bulk-load class.

## How to read this

Each item was verified adversarially: the reported behavior was reproduced or refuted against the real 0.17.1 tree, with concrete file:line evidence cited rather than taken on faith. Competitor handling (Neo4j, Memgraph, Kuzu, and for storage items the object-store peers Delta/Iceberg/SlateDB) was researched to set the field standard. Each item then carries a proposed fix, severity (blocker > high > medium > low, weighing silent-correctness and bulk-load impact heavily), effort (S/M/L), and a wave (now/next/later). Verification confidence is recorded per item; all 18 here landed at high confidence, but where a verdict corrects the user's framing (`already-fixed`, `partially-real`, `cannot-reproduce`) that is called out plainly in the verdict line and the body.

## Findings by theme

### Bulk-load / performance

#### [01-unwind-bind] UNWIND ... MATCH (a {id: row.x}) does not propagate the binding  -  Low / S / now  (Verdict: already-fixed)

- What the user hit: `UNWIND list AS row MATCH (a {prop: row.x})` failed at runtime with "binding `row` not bound", forcing N round-trips for bulk edge loads.
- What the code does today: Fixed on 0.17.1. `lower_unwind` introduces the row alias into the lowering scope (`crates/namidb-query/src/plan/lower.rs:1679-1691`), and for an inline map without `_id` the lowerer builds a NodeScan, cross-products it with the carried-in input, and lifts the per-property `build_eq` Filter ABOVE the join so the UNWIND binding is visible (`lower.rs:855-884`). Predicate pushdown deliberately keeps these predicates above the join: the Unwind case puts any term referencing the unwind alias in `stay` (`crates/namidb-query/src/optimize/pushdown.rs:341-355`), and CrossProduct routes a term hitting both sides to `keep_top` (`pushdown.rs:162-188`). The old runtime error (`crates/namidb-query/src/exec/expr.rs:51-54`) only fired with the pre-fix sunk-predicate plan. Passing regression tests cover both read (`unwind_alias_drives_following_match_property_filter`, `crates/namidb-query/tests/exec_match_expand.rs:811-843`) and write paths (`merge_pattern_property_reads_outer_row_binding`, `crates/namidb-query/tests/exec_writes.rs:909-957`); an auditor scratch test for the full multi-endpoint bulk-edge idiom created exactly 2 edges and passed.
- RFC status: `011-predicate-pushdown.md`.
- How others do it: Neo4j, Memgraph, and Kuzu all treat `UNWIND $batch AS row` driving a following MATCH/MERGE as the canonical bulk-load idiom (openCypher query-horizon rule). None ever required the per-row workaround. NamiDB 0.17.1 now conforms.
- Proposed fix: No production change. Pin coverage in `crates/namidb-query`: promote the auditor scratch test into `exec_writes.rs` as `unwind_bulk_edges_match_both_endpoints_then_create` (the only multi-endpoint shape not yet checked in), and add a pushdown plan-shape assertion that the `build_eq` Filter referencing `row` lands above the Unwind/CrossProduct so a future refactor cannot re-sink it.
- Alternatives: Do nothing beyond verification (leaves the user's exact shape unpinned, so a refactor could silently reintroduce the bug). Recommended: do the cheap test-pinning and route the throughput ambition to 02/03, since the UNWIND-via-Cypher write path is still row-at-a-time and the real performance home is the bulk-throughput work.

#### [02-set-plus-map] SET n += {map} not supported (only SET n.prop = value)  -  High / S / now  (Verdict: confirmed-real)

- What the user hit: any query using `SET n += {map}` (or `SET n = {map}`) parses but fails at execution, forcing one `SET n.prop = value` per property in bulk-load scripts.
- What the code does today: confirmed half-wired. The grammar (`crates/namidb-query/src/parser/grammar.rs:591-611`), AST (`parser/ast.rs:167-193`), and lowering (`plan/lower.rs:1903-1918`) all support `+=` (`SetItem::Merge`) and `=` (`SetItem::Replace`), and the value rides through to the executor on `SetOp::Replace/Merge` (`plan/logical.rs:640-668`). But `apply_set` matches both and unconditionally returns `ExecError::Runtime("SET ... = {...} / += {...} land (use SET prop = value forms)")` (`crates/namidb-query/src/exec/writer.rs:961-966`). Only `SetOp::Property` and `SetOp::Labels` execute. The map-spread helpers the fix needs already exist (`runtime_value_to_core` round-trips `Map`, `node_runtime_props_to_core` converts a whole `BTreeMap`, `writer.rs:1604-1634`).
- RFC status: `009-write-clauses.md` (specs `SetOp::Replace/Merge` as designed ops, not implemented in the executor).
- How others do it: Neo4j and Memgraph fully support both, with the critical asymmetry NamiDB must replicate (empty map with `+=` is a no-op, empty map with `=` wipes all properties; `null` value under `+=` removes the property). `SET n += $props` is the standard upsert idiom. Kuzu does NOT support it (same workaround as NamiDB today), but Kuzu has a structural excuse (fixed columnar schema) NamiDB does not.
- Proposed fix: implement the two stubbed arms in `apply_set` (`writer.rs:961-966`), factoring a helper shared with the `SetOp::Property` arm. Evaluate the map; for `Merge`, start from current props and insert/overwrite, removing keys whose value is `null`; for `Replace`, start from empty and insert all. Run the result through `enforce_unique_on_set` and `enforce_notnull_on_create` against the FINAL prop set (so a `Replace` that drops a NOT NULL column is rejected, not silently corrupting). Persist via `upsert_node_with_labels`/`upsert_edge`, increment `properties_set` by entry count, and verify the MERGE `ON CREATE/ON MATCH SET` paths flow through the same arm.
- Alternatives: desugar in the planner into `SetOp::Property` plus `RemoveOp` (rejected: the planner does not know the node's runtime property set, so `Replace` key-dropping and null-removal cannot be done at plan time). Ship `+=` only first (reasonable if time-boxed, ~90% of the value, but both arms share the helper so doing both is barely more work).

#### [03-bulk-ingest] No bulk ingest path (Arrow/Parquet COPY); everything goes through Cypher  -  High / M / now  (Verdict: partially-real)

- What the user hit: no user-facing way to bulk-load a graph; everything goes through per-row Cypher or per-row SDK upserts.
- What the code does today: partially-real. A real Parquet to WriterSession loader exists (`parquet_loader::load_nodes`, re-exported as `load_nodes_from_parquet`, `crates/namidb-storage/src/parquet_loader.rs:84-124`), but it loads NODES ONLY (no `upsert_edge` anywhere in the file) and its only callers are a Criterion bench and its own unit test (`crates/namidb-storage/benches/parquet_ingest.rs:52,175`). No user surface exposes it: the CLI has no import/copy command (`crates/namidb-cli/src/main.rs:36-180`), the HTTP server exposes only `POST /v0/cypher` (`crates/namidb-server/src/lib.rs:286-298`), and the Cypher grammar has no LOAD/COPY/IMPORT keyword. The edge write primitive the fix needs already exists (`WriterSession::upsert_edge`, `crates/namidb-storage/src/ingest.rs:549-570`).
- RFC status: none.
- How others do it: Neo4j ships `neo4j-admin database import` (offline, Parquet since 5.26) as a first-class admin tool separate from the query language. Memgraph has a `LOAD PARQUET` clause run under an import-optimized analytical storage mode (~1M nodes+edges/sec). Kuzu's `COPY ... FROM 'file.parquet'` is the canonical fastest path, covers nodes AND rels, and enforces the hard rule that relationship COPY requires endpoint nodes to already exist. Kuzu is the closest analog.
- Proposed fix: promote the loader in three layers, reusing the existing WriterSession. (1) Storage: add `load_edges` to `parquet_loader.rs` mirroring `load_nodes` but reading src/dst `FixedSizeBinary(16)` columns and calling `upsert_edge`; factor the property-decode loop out of `ingest_batch` so both share it. (2) CLI: add `Cmd::Import` taking repeatable `--nodes <label>=<file>` and `--edges <type>=<file>:<src>:<dst>`, running all node files first then edges, with a single `flush` at the end. (3) HTTP `POST /v0/ingest/parquet` (server pulls from an object-store URI, behind the write-role check) and Python `Client.load_parquet`. Enforce the nodes-first invariant by failing loudly on a dangling edge endpoint.
- Alternatives: a Cypher-native `COPY ... FROM` clause (larger parser/planner/executor surface, strictly additive on top of the storage `load_edges`, defer to a follow-up). A sidecar per-row loader script (rejected: keeps the slow path, does not close the nodes-only gap). A Memgraph-style no-MVCC import mode (premature; the LSM writer already batches via `commit_every`).

### Cypher incompleto

#### [04-all-any] all() / any() list predicates do not exist  -  Medium / S / next  (Verdict: confirmed-real)

- What the user hit: `all(x IN list WHERE pred)` and `any(...)` do not parse; the user migrated ABAC rules to a workaround.
- What the code does today: confirmed-real. There is no quantifier AST node (`crates/namidb-query/src/parser/ast.rs:359-412` has `ListComprehension` and `PatternComprehension` only), no parse path, and unknown function names hit "function `{}` is not supported in v0" (`crates/namidb-query/src/exec/expr.rs:878-882`). `ALL` is a reserved keyword (`parser/lexer.rs:56,166,746`) so `all(...)` fails at the lexer; `any/none/single` parse as generic function calls and break at the inner `WHERE`. The workaround is confirmed working: `size([x IN list WHERE pred])` comparisons, since list comprehensions and `size()` both work (`expr.rs:627-643`, `grammar.rs:1471-1502`). The `ListComprehension` evaluator (`expr.rs:1187-1223`) is the exact template for a quantifier evaluator.
- RFC status: none.
- How others do it: Neo4j, Memgraph, and Kuzu all ship `all/any/none/single(variable IN list WHERE predicate)` as first-class openCypher predicate functions. Apache AGE added them in a single bounded PR, confirming this is standard, well-scoped work, not research. NamiDB is the outlier.
- Proposed fix: add `ExpressionKind::Quantifier` mirroring `ListComprehension` (required predicate, no projection) plus a `QuantifierKind { All, Any, None, Single }`. In `parse_primary`, special-case the `ALL` keyword (disambiguated from `UNION ALL` / `allShortestPaths` by lookahead) and the `any/none/single` identifiers when followed by `( Ident IN`, reusing the comprehension body-parse. Add `eval_quantifier` modeled on `eval_list_comprehension` with short-circuit and openCypher three-valued (Null) semantics. Extend the ~6 exhaustive `ExpressionKind` matches (`walker.rs:3373`, `lower.rs:512,2063`, `optimize/mod.rs:182`, `projection_pushdown.rs:422`, `display.rs:385`).
- Alternatives: desugar to `size([x IN L WHERE p])` comparisons (zero new evaluator, but loses short-circuit and gets Null three-valued logic subtly wrong, a silent-correctness risk for ABAC; stopgap only). Lambda-based `list_filter` like Kuzu (more general, much larger surface, not justified now). The dedicated node is preferred: small, correct, matches the field.

#### [05-varlen-optional] Variable-length paths inside OPTIONAL MATCH not supported  -  Medium / S / now  (Verdict: confirmed-real)

- What the user hit: `OPTIONAL MATCH (a)-[:KNOWS*1..3]->(b)` hard-errors with parser error E008.
- What the code does today: confirmed-real but cosmetic. The guard is an 11-line block in `parse_match_clause` (`crates/namidb-query/src/parser/grammar.rs:325-336`) firing whenever `has_variable_length` is true. The block is purely at the parser layer: `lower_rel_node` threads `optional` and `rel.length` independently into the Expand node with no gate (`crates/namidb-query/src/plan/lower.rs:901-968`), and `execute_expand` runs the variable-length frontier loop and then, only if nothing matched, the OPTIONAL null-pad branch (`crates/namidb-query/src/exec/walker.rs:893-1188`). The once-ambiguous `*0..n` case is already correct: the `min==0` branch emits the source as a hop-0 match (`walker.rs:961`), so a zero-length OPTIONAL always matches. The combination would execute correctly today if the parser allowed it.
- RFC status: `004-cypher-subset.md` (drawback 5 documents the intentional v0 deferral; the cited `*0..n` ambiguity is already resolved by the walker).
- How others do it: Neo4j, Memgraph, and Kuzu all treat OPTIONAL MATCH as fully orthogonal to pattern shape; none has a special case for the var-length combination. Industry consensus on `*0..n` under OPTIONAL matches the walker: the length-0 path always exists, so OPTIONAL never null-pads for a `min==0` pattern. Kuzu's mandatory finite upper bound mirrors NamiDB's existing E002 guard.
- Proposed fix: parser-only. Delete the guard at `grammar.rs:325-336`; no replacement logic is needed because lowering and the executor already handle the combination. Verify E002 `UnboundedVariableLength` still fires for `OPTIONAL MATCH (a)-[*]->(b)` (it is not gated by the optional flag). Flip the test `optional_match_no_variable_length` to assert successful parse. Add exec tests for bounded-match, bounded-no-match (rel/target null, source preserved), and `*0..1` always-matching. Update RFC-004 drawback 5.
- Alternatives: feature flag (pointless: the executor path is already exercised, no new untested surface). Lowering-level validation (strictly worse: re-imposes a block we want gone). All subsumed by the simple deletion.

#### [06-where-label] Label test in WHERE (WHERE n:Label) not supported  -  Medium / S / now  (Verdict: confirmed-real)

- What the user hit: `MATCH (n) WHERE n:Person RETURN n` fails with E001 UnexpectedToken; same for `WHERE NOT n:Person`.
- What the code does today: confirmed-real. There is no label-test expression form: no `HasLabel` variant in `ExpressionKind` (`crates/namidb-query/src/parser/ast.rs:358-412`), no `Token::Colon` arm in the Pratt table (`parser/grammar.rs:1058-1082`), and `parse_postfix` handles only `.` and `[` (`grammar.rs:1194-1258`). A different-syntax workaround works (`WHERE 'Person' IN labels(n)`). Critically the engine already has full internal label-membership plumbing: the synthetic `__label_eq(node, "Label")` pseudo-call is built by in-pattern lowering (`plan/lower.rs:2096-2117`), understood by the cost model, and evaluated with correct Cypher NULL semantics (`crates/namidb-query/src/exec/expr.rs:852-862`). The fix is to make the parser emit that same pseudo-call.
- RFC status: `004-cypher-subset.md`.
- How others do it: Neo4j fully supports the colon predicate plus the full `&|!%` label algebra and `IS LABELED`. Memgraph supports the colon form and `:A:B` conjunction but not `|` disjunction. Kuzu does NOT support the colon predicate (uses `label(n) = 'Person'`), so NamiDB is at parity with Kuzu today and would leapfrog it.
- Proposed fix: parser-only, reusing `__label_eq`. Add a `Token::Colon` arm to `parse_postfix` that calls the existing `parse_label_list` and builds `__label_eq(<lhs>, "Label")` per label, AND-ing multiple labels. Because `parse_postfix` sits under `parse_unary`, `WHERE NOT n:Person` becomes `NOT __label_eq(...)` for free. Factor the `__label_eq` constructor out of `lower.rs::build_label_eq` into a shared helper. Add a regression test that `WHERE n:Person` is not optimized away by `is_synthetic_label_eq` (`optimize/mod.rs:430-447`, which is keyed by alias+label and only strips a filter an Expand already enforces). Scope to single-label and `:A:B`; defer `|!%` and `IS LABELED`.
- Alternatives: a dedicated `HasLabel` AST variant (cleaner, but touches lowering, optimizer, cost model; better as a later refactor when the `|!` algebra is added). Lower to `'Label' IN labels(target)` (works but materializes the full label list per row and loses the per-label selectivity hint). Do nothing (rejected: most common Cypher idiom, ~1 day fix).

#### [07-datetime-noarg] datetime() with no arguments (now) not supported  -  Medium / S / now  (Verdict: partially-real)

- What the user hit: `datetime()` with no args is rejected.
- What the code does today: partially-real and the gap is wider than reported. There is no `datetime`/`date`/`localdatetime`/`duration` arm in `call_scalar_function`; every temporal constructor falls through to "function `datetime` is not supported in v0" (`crates/namidb-query/src/exec/expr.rs:574-882, 878-881`). So not only the no-arg `now` form fails, the ISO-string form fails too. But all downstream plumbing exists: `RuntimeValue::Date(i32)`/`DateTime(i64)` (`exec/value.rs:25-28`), comparison, storage via `runtime_to_core` (`writer.rs:1560-1561`), Bolt mapping (`crates/namidb-bolt/src/mapping.rs:73-81`), and core serde. The `timestamp()` builtin (`expr.rs:867-876`) is a working zero-arg template. The name is lowercased first (`expr.rs:120`), so one arm covers all casings.
- RFC status: `004-cypher-subset.md` (plans ISO-8601 constructors only; the no-arg form is not even in planned scope).
- How others do it: Neo4j, Memgraph, and Kuzu all ship a zero-arg current-instant builtin (`datetime()`, `localDateTime()`, `current_timestamp()` respectively). NamiDB is the outlier in rejecting it entirely.
- Proposed fix: add temporal arms to `call_scalar_function`, modeled on `timestamp()`; `chrono` is already a dependency. `datetime`: empty args returns `DateTime(Utc::now().timestamp_micros())`; one String arg parses RFC3339; `null` returns null. `date`: zero-arg returns days-since-epoch; one String arg parses `NaiveDate`. Optionally alias `localdatetime`. Add a small `parse_iso_datetime` helper. Update RFC-004 and add tests for `RETURN datetime()`, the ISO form, a `CREATE` round-trip, and `datetime(null)`.
- Alternatives: a `now()` alias only (breaks Neo4j-driver parity). Full temporal algebra with durations/timezones (L/XL, far beyond the ask). Map `datetime()` to the `timestamp()` integer (wrong Cypher type, breaks ORDER BY and driver round-trips). The targeted constructor arms are recommended.

#### [11-generic-500] Generic 500s on unsupported features instead of a typed error  -  Medium / S / next  (Verdict: partially-real)

- What the user hit: unsupported features surface as opaque 500s rather than a typed "not supported" error.
- What the code does today: partially-real. The typed target already exists on one path but not the others. HTTP exec arms hardcode `INTERNAL_SERVER_ERROR` with no code, and `ErrorBody` has no `code` field (`crates/namidb-server/src/lib.rs:858-913, 618-621`); parse/plan 400s discard `LowerErrorKind` (`lib.rs:749-800`). `EvalError` carries only message and span, no kind discriminant, so `PatternComprehension`, `EXISTS`, and unknown functions are indistinguishable (`crates/namidb-query/src/exec/expr.rs:167,172,878`). `ExecError` has no `Unsupported` variant (`crates/namidb-query/src/exec/walker.rs:33-49`). On Bolt, `map_lower_err` IS typed (`LowerErrorKind::UnsupportedFeature` to `BackendError::Unsupported`, `crates/namidb-server/src/bolt.rs:478-484`) and maps onward to `Neo.ClientError.Statement.NotSupported` (`crates/namidb-bolt/src/session.rs:122-168`), but `map_exec_err` substring-buckets exec errors into Storage/Eval (`bolt.rs:486-502`), so exec-time not-supported is mislabeled as `ArgumentError`. The typed lane exists; the exec path never reaches it.
- RFC status: `004-cypher-subset.md`.
- How others do it: Neo4j, Memgraph, and Kuzu all return typed, classified error codes for unsupported features rather than a generic server error.
- Proposed fix: thread the typed-error discriminant end to end. Add an `Unsupported` kind to `EvalError`/`ExecError` so exec-time not-supported is distinguishable from a genuine internal bug, route it to `BackendError::Unsupported` on the Bolt path (replacing the `map_exec_err` substring bucketing), and add a `code` field to the HTTP `ErrorBody` so the same classification surfaces over `/v0/cypher` instead of a bare 500.
- Alternatives: keep substring-matching error text (brittle; the existing `map_exec_err` approach, rejected as the long-term answer).

### Correctness / modelo

#### [09-uniqueness] Uniqueness NOT enforced on the Python bulk path; silent duplicates possible  -  High / M / now  (Verdict: partially-real)

- What the user hit: declared unique-property values silently duplicate on the bulk-load path.
- What the code does today: partially-real, and the residual is a genuine silent hole. The original framing is stale for the Cypher path: CREATE, MERGE-create, and SET all enforce via `enforce_unique_on_create`/`enforce_unique_on_set` against the read-your-own-writes overlay, rejecting duplicates with `ExecError::Constraint` (`crates/namidb-query/src/exec/writer.rs:588-622, 629-665`), proven by `create_rejects_duplicate_unique_property` (`writer.rs:1724-1758`). The real gap is the Python low-level bulk API: `upsert_node`, `upsert_node_with_labels`, and `merge_nodes` call storage `WriterSession::upsert_node` directly (`crates/namidb-py/src/lib.rs:87-128, 281-323`), which does id-keyed last-write-wins dedup only and NO property-uniqueness check (`crates/namidb-storage/src/ingest.rs:498-517`). Two node ids with the same declared-unique value commit silently. The user's reported workload used exactly this path. The `PropertyDef::unique` doc comment (`crates/namidb-core/src/schema.rs:99-107`) still wrongly claims the engine never enforces uniqueness.
- RFC status: none (reuses RFC-026 read-your-own-writes overlay).
- How others do it: Neo4j, Memgraph, and Kuzu all enforce uniqueness/PK constraints on the bulk path, not just the interactive path (Kuzu's COPY rejects duplicate PKs by construction; Memgraph rejects at commit time). None has NamiDB's split where the high-level path enforces but the low-level bulk API silently bypasses. The closest analog to NamiDB's gap is a raw LSM put skipping the application constraint layer, which is exactly what `WriterSession::upsert_node` is.
- Proposed fix: route the Python low-level node-writes through the same check. Lift `find_unique_conflict`/`enforce_unique_on_create` out of `writer.rs` into a shared helper taking a schema, an overlay-capable snapshot, and a `(labels, props)` pair. In `namidb-py/src/lib.rs`, have `upsert_node`/`upsert_node_with_labels`/`merge_nodes` fetch the schema and call the helper before staging, returning a `PyErr` on conflict; for `merge_nodes` also guard intra-batch duplicates with a `HashMap<(label,prop), value>`. Fix the stale doc comment in `schema.rs`. Add Rust and Python tests.
- Alternatives: enforce inside `WriterSession::upsert_node` (rejected: makes the lowest-level primitive schema-aware, hurts throughput, breaks the storage-is-dumb separation). Deferred validation at `commit_batch` time (closer to Memgraph, amortizes the check, but loses the per-row error; viable as a later layer). Kuzu-style mandatory single PK (cleanest long-term but a breaking schema/keying change). Mitigate the O(N^2) scan-fallback risk on non-string unique values with an in-batch `HashSet` and, later, a typed property index.

#### [10-double-identity] Double identity: internal id (uuid7) vs user property `id`  -  Low / M / later  (Verdict: already-fixed)

- What the user hit: a business `id` property collided with the engine's internal edge-linking id, corrupting edge linkage ("Bug #1").
- What the code does today: already-fixed in 0.17.1. Three identity concepts are cleanly separated: the internal storage id is a UUIDv7 `NodeId`/`EdgeId` (`crates/namidb-core/src/id.rs:22-36`), edges store endpoints as internal NodeIds never as user properties; the internal id is surfaced via the `_id` sigil and `id()`/`elementId()` (`crates/namidb-query/src/exec/expr.rs:190-208, 576-592`); and plain `id` is an ordinary user property where `MATCH (n {id: ...})` lowers to NodeScan+Filter, NOT NodeById (`crates/namidb-query/src/plan/lower.rs:2161-2195`). The footgun was fixed 2026-05-18 in commit `9b8f2b8` ("free the `id` property, internal NodeId moves to `_id`"), an ancestor of HEAD, with regression tests (`expr.rs:1648-1678`, `lower.rs:2161-2195`). Residual: a business `id` has no automatic index or uniqueness, so `MATCH (n {id: ...})` is a full scan and duplicate business keys are silently allowed.
- RFC status: none.
- How others do it: Neo4j and Memgraph separate internal identity from user properties exactly as NamiDB now does (id() deprecated for elementId(); use a user property for business keys, with a UNIQUE constraint + backing index). Kuzu fuses business key and fast lookup into a mandatory PRIMARY KEY and exposes the internal id under `_id`, the very sigil NamiDB chose. NamiDB's UUIDv7-never-reused `_id` is a stronger internal-id contract than Neo4j/Memgraph (no reuse-after-delete hazard); the only gap is the optional uniqueness+index on the business key.
- Proposed fix: no change for the reported symptom. Forward hardening, all absorbed by other workstreams: (1) extend `optimize/unique_lookup.rs` so `MATCH (n:L {id: $x})` can target a secondary property index (C2 work); (2) enforce UNIQUE on a chosen business key in the writer (the constraints P0); (3) cheap diagnostics now, warning when both `id` and `_id` appear in one map literal or when a literal `id` looks like a UUIDv7. Add a one-line doc note: `_id` = engine identity, `id` = your property.
- Alternatives: Kuzu-style mandatory PRIMARY KEY (breaks schema-optional positioning). Auto-index every `id` heuristically (surprising, silent index growth). Documentation-only (closes nothing structural). Recommended: ship the cheap diagnostics opportunistically, fold indexing/uniqueness into C2 + constraints.

### Vectores / busqueda

#### [12-flat-vector] Flat (brute-force) vector scan, no ANN; forces external Qdrant at scale  -  High / L / later  (Verdict: confirmed-real)

- What the user hit: vector search latency grows linearly with corpus size, pushing users to an external Qdrant at million-vector scale.
- What the code does today: confirmed-real. Vector search is an exact brute-force linear scan: `cosine_similarity`/`dot_product`/`euclidean_distance` are per-row scalar builtins (`crates/namidb-query/src/exec/expr.rs:808-848`) evaluated over every candidate, and the MCP `vector_search` tool lowers to `MATCH (n:Label) WHERE ... ORDER BY cosine_similarity(...) DESC LIMIT k` (`crates/namidb-mcp/src/lib.rs:450-456`). The only acceleration is a bounded top-k max-heap (`exec/walker.rs:2209-2249`), O(n log k) but still visiting every row and exact, not approximate. No HNSW/IVF exists; the only storage index is a scalar exact-match HashMap (`crates/namidb-storage/src/property_index.rs:1-51`), and the manifest knows only Nodes/EdgesFwd/EdgesInv plus unique/equality/label sidecars (`crates/namidb-storage/src/manifest.rs:116-318`). Shipped int8 quantization (`crates/namidb-core/src/quantize.rs`) shrinks bytes but does not prune. README states verbatim "The scan is still flat (no ANN index yet)".
- RFC status: none (`docs/rfc/001-storage-engine.md:271-273` lists "Vector index integration (Lance v2)" as deferred).
- How others do it: Neo4j (native HNSW since 5.13/5.15), Memgraph (USearch HNSW, on by default since 3.2), and Kuzu (native disk-based HNSW, GA 0.9.0) all ship native ANN. Kuzu is the key analog: its HNSW lower layer is stored AS A RELATIONSHIP TABLE, i.e. the same CSR on-disk structure NamiDB already uses for edges, so the "HNSW graph = CSR edge table" mapping is a near-direct fit. Object-store precedent: LanceDB/Lance stores IVF-PQ and HNSW as immutable manifest-tracked files; DiskANN/Vamana is the canonical object-store-friendly ANN.
- Proposed fix: add an on-disk HNSW index as a new manifest-tracked SST kind, reusing the CSR adjacency machinery (the Kuzu/NaviX model). Add `SstKind::VectorIndex` + a `VectorIndexDescriptor` to `manifest.rs`; persist HNSW layers as a CSR block (reuse the edges writer) plus int8 vector codes (reuse `quantize.rs`). Build during flush/compaction (rebuild-at-compaction avoids in-place graph mutation). Add `CREATE VECTOR INDEX` DDL, a `LogicalPlan::VectorSearch`, and an optimizer pass modeled on `unique_lookup.rs` that rewrites the scan+order+limit shape to a `VectorSearch` operator when a descriptor exists (else falls through to the exact flat path, preserving correctness). Use Kuzu-style prefiltering for WHERE predicates. Gate behind `NAMIDB_VECTOR_INDEX`; validate recall@k via the existing `namidb-bench/vector_recall` harness.
- Alternatives: IVF-Flat/IVF-PQ (simpler, more incremental, lower recall; a viable M-effort v0 if HNSW-on-CSR proves fiddly). Embed `usearch`/`hnsw_rs` as an opaque sidecar blob (fastest to ship, but a black box outside the LSM lifecycle and a C++ dependency cutting against pure-Rust). DiskANN/Vamana (best object-store memory profile, larger from-scratch build). Do nothing / document Qdrant offload (concedes the single-store pitch exactly where it matters).

#### [13-hybrid-search] No hybrid BM25+dense search, no reranking  -  Medium / M / next  (Verdict: confirmed-real)

- What the user hit: no hybrid lexical+dense retrieval and no reranking for the RAG/agent workload.
- What the code does today: confirmed-real. There are exactly two retrieval paths and neither is hybrid or reranked. Lexical is plain substring: Cypher exposes only CONTAINS/STARTS WITH/ENDS WITH (`crates/namidb-query/src/parser/lexer.rs:753-755`) and the MCP `search` tool runs `WHERE n.body CONTAINS $text OR ... LIMIT 100` with no index, scoring, or ranking (`crates/namidb-mcp/src/lib.rs:304-317`). Dense is pure cosine KNN via the vector builtins and the MCP `vector_search` tool (`crates/namidb-query/src/exec/expr.rs:819-848`, `mcp/lib.rs:401-462`). A workspace grep for bm25/tantivy/lucene/rrf/rerank returned zero hits. The default `HashingEmbedder` (`crates/namidb-markdown/src/embed.rs:84-97`) is signed feature-hashing into a dense vector scored by cosine, term-presence only (no TF/IDF/length norm), so it is not a BM25 index and there is nothing to fuse it with.
- RFC status: none.
- How others do it: Memgraph is the closest match (Tantivy-backed BM25 + USearch HNSW side by side in an embeddable Rust engine; fusion left to the query). Neo4j ships Lucene BM25 full-text + HNSW vectors, RRF computed by hand. Kuzu has Okapi-BM25 FTS + HNSW with hybrid fusion explicitly on the roadmap. Ecosystem consensus on fusion is Reciprocal Rank Fusion (RRF, `1/(k+rank)`, k~60), used by Elasticsearch/OpenSearch/Pinecone/Weaviate/Qdrant because it fuses by rank and sidesteps comparing unbounded BM25 to bounded cosine.
- Proposed fix: two layers, smallest useful first. Layer A (now, MCP-only, no engine change): add a `hybrid_search` tool that runs the existing lexical and cosine-KNN channels each at `LIMIT 3*k` and fuses in Rust via `rrf_fuse(channels, k_const=60, weights)` keyed on `path`. Layer A+ (now): add a `bm25(text, query)` builtin in `expr.rs` near `cosine_similarity`, reusing the `tokenize` logic from `embed.rs`, so the lexical channel ranks meaningfully before RRF. Layer B (later, real lexical quality, matches Memgraph): a `namidb-fts` crate wrapping `tantivy`, write-path index maintenance, `CREATE FULLTEXT INDEX` / a `fulltext_search` tool, behind a cargo feature; optional final cross-encoder rerank reusing `crate::remote`.
- Alternatives: sparse-vector-only hybrid reusing the vector machinery (avoids the Tantivy dependency but still no IDF/length norm or phrase queries; papers over the gap). Pure client-side RRF (zero engine work but every consumer reinvents fusion and the lexical channel stays substring-only). Weighted linear score fusion (one knob but requires per-corpus normalization of incompatible ranges; RRF is the safer rank-based default).

#### [08-call-show-algos] No CALL/SHOW/APOC/GDS; no native graph algorithms  -  Medium / L / next  (Verdict: partially-real)

- What the user hit: cannot run native graph algorithms (connected components, PageRank, communities); had to reimplement them in the app.
- What the code does today: partially-real. Cypher has no CALL/SHOW/YIELD clause (`crates/namidb-query/src/parser/lexer.rs:721-765`, `grammar.rs:276-307` reject everything else) and ships zero algorithms. The only "procedure" surface is a Bolt-only string-matched introspection shim that fakes a fixed allowlist of schema procedures so a GUI sidebar populates (`crates/namidb-server/src/introspect.rs:1-92`, wired only on the Bolt path, `bolt.rs:317-374`); it runs no algorithm and HTTP/CLI/Python never see it. The only real algorithm is shortestPath/allShortestPaths, a MATCH-pattern primitive over the variable-length BFS Expand, not a procedure (`grammar.rs:720-747`, `logical.rs:18-130`). The `namidb-graph` crate is an empty `lib.rs:1-9`. The data plane exists: `AdjacencyCache` + `Snapshot::out_edges` over CSR SSTs (`crates/namidb-storage/src/lib.rs:16-40`).
- RFC status: `023-shortest-path.md`.
- How others do it: Neo4j gates PageRank/WCC/Louvain behind the GDS plugin (project a named graph, then `CALL gds.pageRank.stream`). Memgraph ships MAGE modules invoked over `CALL` directly on the live graph (closest model). Kuzu added an opt-in `algo` extension running over a projected graph (the best analog: embeddable engine, algorithms over native adjacency, not an always-on server plugin). All converge on a CALL/YIELD subsystem plus a kernel iterating an adjacency; NamiDB already has the adjacency, so the gap is the clause and the kernels.
- Proposed fix: a minimal CALL subsystem plus a first kernel, two PRs behind `NAMIDB_PROCEDURES`. PR1: add CALL/YIELD to the lexer/AST/grammar, lower to a `CallProcedure` logical node, and add a source operator in the executor that resolves the proc name against a `ProcRegistry` and streams YIELD columns. PR2: fill `namidb-graph` with an `algo` module over `AdjacencyCache`/`Snapshot::out_edges` computing WCC (union-find, cheapest/most useful first) then PageRank (power iteration), registered under `algo.*` plus Memgraph aliases so existing scripts run. Wire the same registry into `namidb-mcp` as a `graph_algorithm` tool since MCP is the product wedge.
- Alternatives: expand the introspect shim to run a kernel (Bolt-only, no args/YIELD composition, string-match debt; a 1-day demo, wrong as the answer). MCP-only kernels with no Cypher CALL (serves the agent wedge immediately, smallest credible slice, gives up Neo4j/Memgraph compat; likely the right FIRST increment with the CALL clause as fast-follow). Full GDS-style projection + write modes up front (XL, premature). Long-running PageRank needs the query-timeout P0 to avoid runaway scans.

#### [14-single-writer] One writer per namespace + one namespace per process  -  High / L / next  (Verdict: confirmed-real)

- What the user hit: serving N tenants requires N OS processes.
- What the code does today: both halves true, but the limit lives entirely in the frontends. One writer per namespace: `WriterSession::open` calls `claim_writer` which CAS-bumps the manifest epoch, and every commit calls `fence.assert_alive` (`crates/namidb-storage/src/fence.rs:43-68`, `manifest.rs:693-748`), so a second writer is structurally fenced (RFC-001 explicit non-goal). One namespace per process: `NamespacePaths` holds exactly one `NamespaceId` (`crates/namidb-storage/src/paths.rs:30-49`), and the server builds ONE `AppState` with a single namespace and a single `Arc<Mutex<WriterSession>>` with fixed routes and no namespace param (`crates/namidb-server/src/lib.rs:117-182, 286-298`); same for CLI and MCP. Key finding: the storage layer is already cleanly per-namespace and N WriterSessions can share one `Arc<dyn ObjectStore>`; the single-namespace limit is purely frontend.
- RFC status: `001-storage-engine.md` (drawback #2; single-writer is a deliberate non-goal, NOT changed by this fix).
- How others do it: Neo4j (4.0+) and Memgraph Enterprise both host N databases in ONE process with a router (`:use <db>` / `USE DATABASE`), which is exactly the in-process namespace router NamiDB lacks. Kuzu is the single-writer peer (no router; open N Database handles in-process). The single-writer-per-namespace half is the RIGHT default and matches Delta/Iceberg/SlateDB (per-table serialization via manifest CAS). The reference for fixing only the frontend is Neon: stateless compute over shared multi-tenant S3 storage; NamiDB Cloud is described in exactly those terms.
- Proposed fix: keep single-writer-per-namespace; fix only the frontend with an in-process `NamespaceRegistry { store, root, sessions: RwLock<HashMap<NamespaceId, Arc<NamespaceState>>> }`. Split `AppState` so process-wide pieces (auth, metrics, timeouts) are shared and per-namespace pieces move into `NamespaceState`; add `get_or_open(ns)` that lazily opens a WriterSession (storage already supports N over one store), with an LRU + idle-eviction cap. Add the namespace to the route (path param or header) with a `--default-namespace` back-compat. Move the per-namespace flush/compaction/orphan-sweep tasks into `NamespaceState` (the main real work). Mirror in MCP; extend `AuthConfig` for per-namespace token scoping. Gate behind `NAMIDB_MULTI_NS`.
- Alternatives: N-processes behind a reverse-proxy that spawns/reaps per-tenant processes (strongest isolation, zero engine change, but heavy ops and essentially rebuilds NamiDB Cloud's closed-source wedge in the open). In-process registry plus DROP single-writer for multi-master CRDT writes (the RFC-001 v2 open question; large research-grade change, unnecessary for the N-isolated-tenants case the user described). The proposed fix keeps single-writer and only multiplexes namespaces, the cheap correct 80%.

#### [15-auth-rbac] Security only by namespace; single bearer token; no RBAC/OIDC, no row/property-level security  -  Medium / L / next  (Verdict: partially-real)

- What the user hit: needed an external OPA gateway for ABAC and row/property-level policy.
- What the code does today: partially-real. Authorization is namespace-coarse with two static-token roles (ReadOnly/ReadWrite) compared in constant time (`crates/namidb-server/src/auth.rs:1-33, 54-141`); the only authorization decision beyond authentication is `role.allows_write()` gating write Cypher and admin flush (`crates/namidb-server/src/lib.rs:803-819, 923-933`). There is no OIDC/JWT/SAML/LDAP, no named roles/permissions/principals/grants, and no row/property/label-level security anywhere. The Bolt path reuses the same static `AuthConfig` (`crates/namidb-server/src/bolt.rs:526-553`). The MCP server is an unauthenticated single-user local stdio process by design (`crates/namidb-mcp/src/lib.rs:1-17`). The only overstatement is "a single token with no roles": `auth.rs` already supports multiple file-loaded tokens each scoped RO or RW.
- RFC status: none (the Bolt doc's "JWT lands with RFC-024" note is stale; RFC-024 is WCOJ).
- How others do it: Neo4j Enterprise is the gold standard (native RBAC with per-label/per-relationship/per-property GRANT/DENY, property-based row-level access, OIDC + SAML + LDAP). Memgraph Enterprise has RBAC + label-based access control + OIDC/SAML SSO, but no native property/row-level. Kuzu is embedded with no RBAC/OIDC (auth lives in the embedding app), matching NamiDB's MCP-over-stdio posture. The pragmatic field split: ship native authentication + coarse RBAC in the engine and externalize fine-grained ABAC to a PDP (OPA/Cedar/OpenFGA), which is the OPA gateway the user built; NamiDB currently exposes neither a verifiable principal nor a policy hook.
- Proposed fix: two waves, contained to the 4-file auth surface, RFC-028 first. Wave A (OIDC/JWT): generalize the resolver to an `Authenticator` enum (`StaticTokens` | `Jwt`), add a feature-gated `jwt` module (`jsonwebtoken` + JWKS fetch/cache, pinned algorithms) verifying signature/iss/aud/exp and mapping group claims to roles, and thread a `Principal` (not just `Role`) into request extensions on both HTTP and Bolt; add `--oidc-*` flags. Wave B (named roles + PDP hook): replace the two-variant `Role` with a named-role model (keeping `allows_write()` as a shim), and add an `AuthzHook` trait called after planning with `PlanContext { labels_touched, properties_read, write }`, default-allow, with an optional `ExternalPdp` impl that POSTs to OPA/Cedar so the user's gateway moves in-process as a decision call (planner already knows labels/properties touched via RFC-025 stats).
- Alternatives: authentication-only, defer all RBAC (closes the most-cited SSO gap, leaves row/property external; good "now" slice). Native fine-grained GRANT/DENY parity with Neo4j (most powerful, self-contained, but XL and deep into the planner/executor, wrong fit pre-adoption). Pure delegation: formalize the OPA-sidecar contract (cheapest, matches the embedded philosophy, concedes the enterprise checkbox).

#### [16-maturity] Alpha/BSL; backup/restore maturing; only the Python Bolt driver verified  -  Medium / M / next  (Verdict: partially-real)

- What the user hit: alpha/BSL maturity, immature backup/restore, only the Python driver verified.
- What the code does today: all three sub-claims hold, but "backup/restore maturing" understates it. `backup.rs` ships BOTH backup and restore as one primitive `copy_namespace_snapshot` (`crates/namidb-storage/src/backup.rs:77-186`), exported and wired into `namidb backup`/`namidb restore` (`crates/namidb-cli/src/main.rs:147-179, 261-309`), with 5 passing tests covering flushed-SST + unflushed-WAL round-trip, self-contained renumbering, no-clobber, side-car copying, and empty namespace (`backup.rs:208-523`). The gaps justify "maturing": the no-clobber guard is a best-effort `head()` check with a TOCTOU race, not a CAS (`backup.rs:85-92`); the destination pointer is written with unconditional `PutMode::Overwrite` (`backup.rs:150-178`) even though the conditional CAS primitives already exist and are used by the commit path (`manifest.rs:646-654, 750-755`); restore resets to `Epoch::ZERO` instead of fencing stale writers; no post-restore consistency verify; no RFC. The driver claim is exactly right: the only driver test is `tests/bolt_neo4j_driver_smoke.py:1-40` (official Python driver); README:99 says other drivers "are not all exercised yet". Alpha/BSL is accurate (`README.md:76-82`).
- RFC status: none (this issue should create one).
- How others do it: Neo4j ships production online backup with differential/incremental, an integrated consistency check, point-in-time restore, and cloud targets. Memgraph's durability snapshots double as the backup artifact with an online locked-snapshot primitive. Kuzu has no online backup daemon (logical EXPORT/IMPORT, or copy the closed file). The deeper analogy is object-store table formats: a NamiDB backup IS a copy of an immutable object closure then a single pointer flip, exactly Iceberg's metadata-pointer CAS and Delta's `_delta_log` PUT-if-absent. CRITICAL: NamiDB already uses these conditional primitives in its commit path but `backup.rs` does not, so the fix is to reuse machinery the engine already trusts.
- Proposed fix: harden the existing primitive, do not rewrite. (1) Make the no-clobber guard a real CAS: replace the final pointer `PutMode::Overwrite` + `head()` pre-check with `PutMode::Create` (If-None-Match:*) when not overwriting, and a read-etag-then-`PutMode::Update` CAS when overwriting, mirroring `manifest.rs`. (2) Add a post-copy consistency verify (a `--verify` flag) that re-opens the destination and head()s every SST and WAL object with the expected size. (3) Fence the restore destination by setting the restored epoch to `current.next()` instead of `Epoch::ZERO`, so a stale writer trips on its next commit. (4) Write the missing RFC stating the consistency-by-construction argument and explicit non-goals (no differential, no PITR), referenced from README.
- Alternatives: differential/incremental backup (cheap given immutable content-addressed objects, but adds a chain/retention/GC model; defer). Point-in-time restore via retained manifest versions (fights the orphan-sweep horizon; large). A true FREEZE/quiescence primitive (cleaner operator story, couples backup to the live server lifecycle and the compaction scheduler A1; medium-large). Driver-coverage matrix in CI (orthogonal, cheap per driver). Object-store backends differ in conditional-write support, so the CAS path needs a capability check or documented fallback for LocalFileSystem.

### Storage portability (S3 conditional put)

#### [s3a-conditional-put] S3 backend never calls with_conditional_put  -  Low / S / later  (Verdict: cannot-reproduce)

- What the user hit: belief that the S3 backend does not enable conditional puts, breaking manifest CAS on non-AWS endpoints.
- What the code does today: cannot-reproduce against the pinned dependency. `object_store` resolves to 0.13.2 (Cargo.lock), whose S3 backend defaults to `S3ConditionalPut::ETagMatch`, emitting standard If-Match (`PutMode::Update`) and If-None-Match:* (`PutMode::Create`) endpoint-independently, which is exactly what NamiDB's manifest CAS depends on (`crates/namidb-storage/src/manifest.rs:646-661`). `AmazonS3Builder::from_env()` at `crates/namidb-storage/src/uri.rs:189-198` never calls `with_conditional_put`, but the field resolves to the `ETagMatch` default when unset, so Nutanix Objects, R2, and MinIO get conditional puts by default; `AWS_CONDITIONAL_PUT=etag` only re-asserts that default. The real (separate, latent) risk: the dependency is implicit, so an explicit `AWS_CONDITIONAL_PUT=disabled` (honored by `from_env`) or an `object_store` major-version default change would silently disable CAS with no preflight error, and Garage/SeaweedFS/RustFS do not advertise precondition support (documented in `crates/namidb-storage/tests/s3_integration.rs:3-21`).
- RFC status: `001-storage-engine.md`.
- How others do it: not applicable to Neo4j/Memgraph/Kuzu (none is object-store-as-database). The relevant peers are Delta (If-None-Match PUT-if-absent, DynamoDB LogStore fallback before S3 added the primitive in Aug 2024), Iceberg (catalog CAS), and SlateDB (If-None-Match manifest CAS). NamiDB's primitive choice is industry-standard and the default config is correct; the only gap vs mature systems is the absence of a fallback path and a preflight capability check.
- Proposed fix: treat as hardening + observability. (1) Pin the dependency explicitly: call `.with_conditional_put(S3ConditionalPut::ETagMatch)` in `parse_s3` so NamiDB requires the mode regardless of the default or any ambient env, neutralizing `AWS_CONDITIONAL_PUT=disabled`. (2) Add an `s3://` `conditional_put=etag|dynamo:<table>|disabled` query param (mirroring object_store's config grammar) for the Garage/SeaweedFS case and the DynamoDB fallback. (3) Add a one-shot CAS preflight in `ManifestStore` init: a conditional Create of a sentinel key; if the backend returns NotImplemented/NotSupported, fail fast with a clear error instead of silently losing CAS at commit time. Extend `s3_integration.rs`.
- Alternatives: docs-only (cheapest, leaves the silent-failure surface as confusing commit-time errors). Pin the mode only, skip preflight (removes the env-override footgun but Garage still fails opaquely). Full DynamoDB LogStore fallback (highest portability, pulls in a Dynamo dependency, defer until a real customer needs a non-conditional endpoint). Recommended: ship the explicit pin + loud preflight, expose the escape-hatch param, defer the Dynamo path.

#### [s3b-versioned-pointer] Versioned pointer depending ONLY on If-None-Match:* instead of If-Match overwrite  -  High / M / next  (Verdict: confirmed-real)

- What the user hit: proposal to replace the If-Match pointer overwrite with a Create-only versioned pointer family so NamiDB depends on a single, universally supported conditional primitive.
- What the code does today: confirmed-real and unimplemented. `cas_pointer` advances the pointer with `PutMode::Update(UpdateVersion{...})`, an If-Match overwrite of `current.json`, and this is the ONLY production consumer of `PutMode::Update` in the storage crate (`crates/namidb-storage/src/manifest.rs:639-691`). It is the sole reason `LocalFileObjectStore` exists (an `flock` advisory-lock workaround because LocalFileSystem returns NotImplemented for Update, `crates/namidb-storage/src/local.rs:4-11, 140-194`). The proposed Create-only family (`pointer/v<N>.json` written via If-None-Match:*, read by LIST/max-N) does not exist and contradicts RFC-001's documented two-primitive design (`docs/rfc/001-storage-engine.md:128-155`). The `ManifestPointer` body can be kept verbatim (`manifest.rs:102-109`); `WriterFence` already provides epoch fencing (`fence.rs:41-68`); the janitor already GCs a monotonic version family (`janitor.rs:40-44`).
- RFC status: none (new RFC needed, amending RFC-001 step-5).
- How others do it: not comparable for Neo4j/Memgraph/Kuzu (no remote mutable pointer). The field has converged on exactly this proposal. Delta commits `NNNN.json` PUT-if-absent, pointer = highest N from a LIST (and moved to pure If-None-Match once S3 added it in Aug 2024). Iceberg takes the other fork (external catalog CAS). SlateDB is the closest twin: manifest files with If-None-Match CAS ("first to create wins") plus a monotonic writer epoch for fencing, i.e. exactly Create-only-pointer + epoch fence, adopted because S3 only guarantees If-None-Match. Primitive matrix (verified 2026-06): If-None-Match:* is universal (S3, GCS, Azure, R2, MinIO, Tigris, Nutanix); If-Match for PUT is the spotty one and is exactly what LocalFileSystem refuses. The user's portability argument is correct.
- Proposed fix: make the pointer a Create-only monotonic family inside `ManifestStore`, keeping `LoadedManifest`/`ManifestPointer` stable so flush/compact/ingest need no edits. Add `pointer_version(N)` (zero-padded hex) to `paths.rs`; bootstrap writes `p0.json` via Create; `load_current` LISTs the pointer dir, takes max-N, GETs it (with an N+1 HEAD guard for eventually-consistent LIST); `cas_pointer` replaces `PutMode::Update` with `put_create(pointer_version(new.version))`, mapping AlreadyExists to the existing Fenced/ManifestCommitCas discrimination, collapsing body-CAS and pointer-CAS onto ONE primitive. Extend the janitor to GC `p<N>.json` below the horizon. Once nothing uses `PutMode::Update`, mark `LocalFileObjectStore::put_with_cas` `#[cfg(test)]` and remove the flock machinery from the hot path. Write the RFC amending RFC-001 step-5.
- Alternatives: Iceberg/catalog model (regains strong serialization with no object-store primitive, but adds an external dependency and breaks "the bucket IS the database"). Keep `current.json` as an advisory cached hint over the authoritative Create-only family (saves the LIST on hot reads; a later optimization on top). Bounded-LIST/gallop with remembered last-N (pure perf, fold into load_current). Do nothing and bless `LocalFileObjectStore` (cheapest, but keeps If-Match on the commit hot path and the advisory-lock workaround load-bearing).

| later | s3a-conditional-put | low | S | explicit `with_conditional_put` pin + loud CAS preflight | non-conditional-endpoint safety (resolved by s3b) |

Recommended sequence: clear the now wave first because it is dominated by the user's most painful bulk-load blockers and the one silent-correctness hole. Ship `02-set-plus-map` and `03-bulk-ingest` together (they share the bulk-load surface and `02` is the canonical upsert idiom), then `09-uniqueness` so the Python bulk path stops silently committing duplicates; the remaining now items (`01`, `05`, `06`, `07`, `10`-diagnostics) are S-effort, near-zero-risk Cypher-parity and test-pinning wins that ride along cheaply. In the next wave, lead with `s3b-versioned-pointer`: it collapses the commit path onto a single universally-supported conditional primitive (If-None-Match:*), retires the `LocalFileObjectStore` advisory-lock workaround, and is the precondition for credibly claiming GCS/R2/Nutanix/MinIO portability; because it resolves the root cause `s3a` documents, landing it dissolves `s3a` (which can then be closed or downgraded to the cheap explicit-pin slice). Sequence `14-single-writer` right after, since the storage layer is already per-namespace clean and the registry unblocks the multi-tenant story. Defer `12-flat-vector` to later as the flagship next-major-capability: it is an L-effort net-new index subsystem with recall/maintenance unknowns and its own RFC, and the CSR + int8 reuse de-risks it enough to be the strongest candidate to promote to next once the painful and correctness items clear.

## Cross-cutting themes

- Half-wired features dominate the cheap wins: `02-set-plus-map`, `06-where-label`, and `07-datetime-noarg` all have the front half (parser/AST/lowering) and the downstream machinery (`__label_eq`, the DateTime value pipeline, the map-spread helpers) already built and tested; only one executor or parser arm is missing. The pattern is "wire the existing half", not "design new machinery", which is why several high/medium items land at S effort.
- Constraint enforcement vs planner hints is a recurring fault line: uniqueness is enforced on the Cypher path but bypassed on the Python bulk path (`09`), and a business `id` carries no index or uniqueness (`10`). The fix in both cases is to make the bulk/low-level path call the same enforcement layer the query path already trusts, keeping storage primitives schema-unaware.
- Silent-correctness outranks loud failure: the items that earn "now" on severity (`09`) are the ones that commit wrong data with no error, while loud parse-time errors (`04`, `05`, `06`) are medium even when the idiom is common. A parse error tells the user; a silent duplicate corrupts the graph invisibly.
- Typed errors are a thread of their own (`11`): the engine has a typed "not supported" lane on the Bolt path but discards the discriminant on the exec and HTTP paths. Threading one `Unsupported` kind end to end is small and makes every transport classify failures consistently.
- The single conditional primitive is the storage strategy: `s3a` confirms the If-Match default is already correct, and `s3b` collapses the commit path onto If-None-Match:* alone, which is what makes "the bucket is the database on any S3-compatible store" true and retires the `LocalFileObjectStore` workaround. The whole storage-portability story reduces to depending on PUT-if-absent and nothing rarer.
- ANN/hybrid-search is one strategic bet, not two: `12-flat-vector` (HNSW on CSR) and `13-hybrid-search` (BM25 + RRF) together are the retrieval roadmap for the MCP/agent wedge. Neither corrupts data or blocks bulk-load, both have tractable first slices (IVF/MCP-only kernel for `12`, the RRF tool for `13`) that de-risk the larger index subsystems behind them.
